PART 1: Sample job with step data — illustrating schema and relationships
================================================================================

Data sources:
  create_job_table columns:
    - job_db_inx: internal DB key (links to steps)
    - id_job: Slurm job ID (what users see)
    - cpus_req: requested CPUs
    - tres_req/tres_alloc: TRES-encoded resource strings
    - time_submit/time_start/time_end: Unix timestamps
    - timelimit: requested time limit (minutes)

  create_step_table columns:
    - job_db_inx: FK to job table
    - id_step: step ID (<0 = special, >=0 = srun steps)
    - step_name: human-readable name
    - user_sec/user_usec: user-mode CPU time (seconds + microseconds)
    - sys_sec/sys_usec: system-mode CPU time (seconds + microseconds)
    - tres_usage_in_max: peak resource usage (TRES-encoded)

PART 5: Distribution of batch CPU time in multi-step jobs
================================================================================

For jobs with both batch and srun steps, how much CPU does batch use?
This helps determine if batch is overhead or real work.

  batch_cpu_range     job_count         %
  ---------------  ------------  --------
         0 (zero)        191119    88.29%
         1-10 sec         14510     6.70%
       11-100 sec          4784     2.21%
     101-1000 sec          1899     0.88%
        >1000 sec          4163     1.92%

  Total multi-step jobs: 216,475

Interpretation:
  - If most jobs show batch_cpu in 0-10 sec range, batch is just shell overhead
  - If batch_cpu is substantial, the job may be doing work in the batch script
    rather than through srun (which is fine, but affects how we sum CPU time)
