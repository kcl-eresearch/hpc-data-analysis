SUMMARY
================================================================================

This script verifies that our MySQL/TRES-based extraction pipeline
(slurm_utils.py) produces the same job-level metrics as sacct.

Key findings:
  - tres_req TRES ID 2 stores total requested memory in MB
  - tres_usage_in_max TRES ID 2 stores peak memory usage in bytes
  - The MB-to-bytes conversion (× 1024²) is correct
  - Both --mem-per-cpu and --mem (per-node) jobs are handled correctly
    because tres_req already contains the TOTAL (Slurm pre-multiplies
    per-cpu values by cpus_req)
  - TotalCPU: 47 jobs (0.06%) where sacct
    reports 00:00:00 but our DB pipeline found actual CPU time by summing
    step-level data directly. This suggests sacct's job-level TotalCPU
    field sometimes fails to aggregate step data, while our pipeline
    sums it from the step table (create_step_table) and is more complete.
